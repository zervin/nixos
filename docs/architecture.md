# NixOS System Architecture

## Overview

This document describes the proposed NixOS system architecture to replicate and improve upon the current Debian 13 KDE Plasma setup on a Framework 13 AMD 7840U laptop. The architecture uses **Nix Flakes** as the configuration entry point, **Home Manager** for user-level configuration, and a modular file layout for maintainability.

## Why Flakes (Not Channels)

Flakes are the recommended modern approach for NixOS configuration:

- **Reproducibility**: `flake.lock` pins exact nixpkgs commits. You get the same build every time.
- **Composability**: Pull in external inputs (nixos-hardware, home-manager, nix-flatpak) declaratively.
- **No channel management**: No `nix-channel --update` dance. Just `nix flake update` to bump everything.
- **Community standard**: Most NixOS configs, templates, and guides assume flakes in 2025+.

## Channel Strategy: Stable Base + Selective Unstable Overlay

The system uses **stable nixpkgs** as the base with a **secondary unstable input** for specific packages that need newer versions. This gives the reliability of stable with escape hatches where it matters.

### Why This Pattern (Not Full Unstable)

| Approach | Stability | Freshness | Complexity |
|---|---|---|---|
| Full nixos-24.11 stable | High | Frozen at release | Low |
| Full nixos-unstable | Good (CI-gated) | Days behind upstream | Low |
| **Stable + unstable overlay** | **High for 95% of system** | **Selective freshness** | **Medium** |

The stable + overlay pattern is the right choice here because:

- Work-critical apps (Citrix, Zoom, ZoomVDI) need to work reliably. Stable reduces the surface area for breakage.
- Citrix Workspace has a history of version lag and dependency issues (libwebkitgtk2 regressions) on stable channels. Pulling it from unstable solves this without destabilizing everything else.
- The base system (kernel, KDE, systemd, Mesa) stays on the stable train with 6-month release cycles.
- When nixos-25.05 ships (~May 2026), updating the stable input is a one-line change in `flake.nix`.

### How the Overlay Works

In `flake.nix`, two nixpkgs inputs are declared:

```nix
inputs = {
  # Primary: stable channel (start on 24.11, upgrade to 25.05 when released)
  nixpkgs.url = "github:NixOS/nixpkgs/nixos-24.11";

  # Secondary: unstable for specific packages
  nixpkgs-unstable.url = "github:NixOS/nixpkgs/nixos-unstable";
};
```

In the system configuration, the unstable input is made available as an overlay:

```nix
# In configuration.nix or a dedicated overlay module
{ config, pkgs, lib, ... }:
let
  # Import unstable nixpkgs with the same config (allowUnfree, etc.)
  pkgs-unstable = import inputs.nixpkgs-unstable {
    system = "x86_64-linux";
    config.allowUnfree = true;
  };
in
{
  # Use pkgs-unstable.citrix_workspace instead of pkgs.citrix_workspace
  environment.systemPackages = [
    pkgs-unstable.citrix_workspace
  ];
}
```

### Packages from Stable vs Unstable

**From stable (nixos-24.11 / 25.05)** -- everything by default:
- KDE Plasma 6, SDDM, Wayland stack
- Kernel, Mesa, Vulkan, AMD GPU drivers
- Docker, Podman, Ollama, Tailscale, Syncthing
- Browsers (Brave, Firefox ESR, Chrome)
- Dev tools (Node.js, Python, Rust, Go, Java)
- Creative apps (GIMP, Inkscape, LibreOffice, Bambu Studio)
- Zoom (`zoom-us`)

**From unstable (selective overlay)** -- only when stable has known issues:
- `citrix_workspace` -- historically version-lagged on stable, libwebkitgtk2 compatibility issues
- Any other package where stable has a critical regression (add as needed)

The key discipline: default to stable. Only add a package to the unstable overlay when you hit a concrete problem with the stable version. Document why in a comment next to the override.

### Upgrading Stable Channels

When nixos-25.05 releases (~May 2026):

```nix
# Change one line in flake.nix:
nixpkgs.url = "github:NixOS/nixpkgs/nixos-25.05";
# Then:
# nix flake update
# sudo nixos-rebuild switch --flake .#framework
```

Home Manager follows the same nixpkgs input, so it upgrades in lockstep.

## Flake Inputs

```nix
{
  inputs = {
    # Primary: stable channel
    nixpkgs.url = "github:NixOS/nixpkgs/nixos-24.11";

    # Secondary: unstable for selective package overrides
    nixpkgs-unstable.url = "github:NixOS/nixpkgs/nixos-unstable";

    # Home Manager (pinned to stable nixpkgs)
    home-manager = {
      url = "github:nix-community/home-manager/release-24.11";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    # Framework laptop hardware support
    nixos-hardware.url = "github:NixOS/nixos-hardware";

    # Declarative disk partitioning (optional, for reproducible installs)
    disko = {
      url = "github:nix-community/disko";
      inputs.nixpkgs.follows = "nixpkgs";
    };

    # Declarative Flatpak management
    nix-flatpak.url = "github:guyber/nix-flatpak";
  };
}
```

Note: Home Manager's `release-24.11` branch tracks the stable nixpkgs branch. When upgrading to nixos-25.05, also update Home Manager to `release-25.05`.

## Module Architecture

### Layer 1: Hardware (system level)

**Framework 13 AMD module** (`nixos-hardware.nixosModules.framework-13-7040-amd`):
- Loads correct AMD CPU microcode, GPU firmware
- Enables fwupd for Framework firmware updates
- Sets up power management (power-profiles-daemon, tlp alternative)
- Configures fingerprint reader (if present)
- Enables ambient light sensor

**GPU module** (custom `modules/hardware/gpu.nix`):
- AMD Radeon 780M via `amdgpu` kernel driver (loaded by default)
- Mesa with Vulkan (RADV), VA-API (hardware video decode)
- 32-bit Mesa libs for Steam/gaming
- OpenCL via ROCm or rusticl for AI workloads (optional)

**Disk layout**:
- GPT with EFI System Partition (~512MB, FAT32, mounted /boot)
- LUKS2 encrypted root partition (remainder of NVMe)
- Inside LUKS: btrfs with subvolumes (@, @home, @nix, @swap)
- zram for swap (recommended for 54GB RAM system)

### Layer 2: Desktop Environment (system level)

#### Dual DE: KDE Plasma 6 + GNOME (Simultaneous)

Both desktop environments are enabled simultaneously. The display manager (SDDM) presents a session picker at login.

**KDE Plasma 6 module** (`modules/desktop/kde.nix`):
```nix
services.desktopManager.plasma6.enable = true;
services.displayManager.sddm = {
  enable = true;
  wayland.enable = true;
};
```

KDE strengths for this user:
- Superior fractional scaling on the Framework 13 display (13.5" at 2256x1504)
- KDE Connect, Plasma Vault, Dolphin file manager
- KDE PIM (Akonadi, Kontact, Merkuro) if desired

**GNOME module** (`modules/desktop/gnome.nix`):
```nix
services.xserver.desktopManager.gnome.enable = true;
# Do NOT enable GDM when SDDM is primary -- use SDDM for both
# GDM and SDDM conflict; pick one display manager

# GNOME Online Accounts -- the key reason for GNOME
services.gnome.gnome-online-accounts.enable = true;

# Evolution for email/calendar/contacts integrated with Online Accounts
environment.systemPackages = with pkgs; [
  gnome-online-accounts
  evolution
  evolution-data-server
  gnome-calendar
  gnome-contacts
];
```

GNOME strengths for this user:
- **GNOME Online Accounts**: Native Google (Drive, Gmail, Calendar) and Microsoft (OneDrive, Outlook) integration
- Evolution email client with full Google/Microsoft account sync
- GNOME Calendar and Contacts sync via Online Accounts
- This replaces Insync for cloud storage when using GNOME sessions

**How the session picker works:**
- SDDM is the display manager for both DEs
- At login, a dropdown lets you choose "Plasma (Wayland)", "GNOME", or "GNOME on Xorg"
- Each session starts its own full DE with appropriate XDG portals
- User data (`/home/zervin`) is shared between sessions
- GNOME and KDE settings coexist in `~/.config/` without conflict (they use different directory names)

**Switching the default session:**
```nix
# SDDM remembers last used session per user. To set an explicit default:
services.displayManager.defaultSession = "plasma";  # or "gnome"
```

**Important: XDG portals for dual DE:**
```nix
xdg.portal = {
  enable = true;
  extraPortals = [
    pkgs.xdg-desktop-portal-kde
    pkgs.xdg-desktop-portal-gnome
  ];
  # Let each DE use its own portal implementation
  config = {
    common = {
      default = [ "kde" "gnome" ];
    };
  };
};
```

#### Cloud Storage Strategy by DE

| DE | Google Drive | OneDrive | Email/Calendar |
|---|---|---|---|
| GNOME | GNOME Online Accounts (native in Files) | GNOME Online Accounts | Evolution |
| KDE | rclone mount or Flatpak Insync | rclone mount or Flatpak Insync | Kontact/Merkuro or Betterbird |

When in GNOME: Google Drive and OneDrive appear as mounted folders in Nautilus (GNOME Files) automatically via GNOME Online Accounts. No Insync needed.

When in KDE: Either use rclone with systemd user service for cloud mounts, or install Insync via Flatpak. KDE's online account integration is less mature.

**Flatpak module** (`modules/desktop/flatpak.nix`):
```nix
services.flatpak.enable = true;
```

With nix-flatpak, you can declaratively manage Flatpak remotes and installed apps:
```nix
services.flatpak.packages = [
  "eu.betterbird.Betterbird"     # Email client
  "net.mkiol.SpeechNote"         # Speech Note
  # Insync only needed for KDE sessions:
  # "com.insynchq.insync"
];
```

## Security Architecture

### NixOS Security Model

NixOS has a fundamentally different security posture from Debian:

**Strengths:**
- Immutable system paths (`/nix/store` is read-only, world-readable but not writable)
- Atomic upgrades and rollbacks (broken update = reboot to previous generation)
- Reproducible builds reduce supply-chain risk
- No "stale package" problem (entire system rebuilt together)
- Declarative firewall rules
- No leftover configs in `/etc` from removed packages

**AppArmor on NixOS:**

NixOS supports AppArmor natively. It is not "less common" -- it is a first-class module:

```nix
security.apparmor = {
  enable = true;
  # Load AppArmor profiles for installed packages
  # NixOS ships profiles for some packages (e.g., Firefox)
  killUnconfinedConfinables = false;
  packages = with pkgs; [
    apparmor-profiles       # Community profiles
  ];
};
```

AppArmor is relevant here for:
- Flatpak sandboxing (Flatpak uses its own sandbox but AppArmor adds defense-in-depth)
- Container workloads (Docker/Podman can use AppArmor profiles)
- Browser confinement (Firefox has upstream AppArmor profiles)

**Audit subsystem:**

```nix
security.audit = {
  enable = true;
  rules = [
    "-a exit,always -F arch=b64 -S execve"  # Log all program executions
  ];
};
security.auditd.enable = true;
```

This is optional but useful for compliance or security-conscious setups. Most desktop NixOS users do not enable it.

**systemd hardening:**

NixOS services can use systemd sandboxing features declaratively:
```nix
systemd.services.some-service = {
  serviceConfig = {
    DynamicUser = true;
    PrivateTmp = true;
    ProtectSystem = "strict";
    ProtectHome = true;
    NoNewPrivileges = true;
  };
};
```

Many NixOS service modules already apply these hardening options by default.

**SELinux:** Not supported on NixOS.

### Recommendation

Enable AppArmor for parity with the Debian setup and container/Flatpak defense-in-depth. The immutable Nix store and atomic rollback provide security guarantees that Debian does not have, making the overall posture at least equivalent even without SELinux.

## Update Workflow

### Day-to-Day Updates

```bash
cd /home/zervin/projects/nixos

# Update stable nixpkgs only:
nix flake update nixpkgs

# Update unstable nixpkgs only (for Citrix etc.):
nix flake update nixpkgs-unstable

# Update everything (both nixpkgs, home-manager, nixos-hardware):
nix flake update

# Build first to check for issues (does not activate):
sudo nixos-rebuild build --flake .#framework

# If build succeeds, switch:
sudo nixos-rebuild switch --flake .#framework

# If something breaks after switching:
sudo nixos-rebuild switch --rollback
# Or select previous generation at boot menu
```

### Adding a New Package

1. Add package to the appropriate module file (e.g., `home/zervin/apps.nix`)
2. Run `sudo nixos-rebuild switch --flake .#framework`
3. Package is installed. Commit the config change to git.

### Adding a Package from Unstable

When a stable package has issues:
1. Add a comment explaining why unstable is needed
2. Reference `pkgs-unstable.<package-name>` instead of `pkgs.<package-name>`
3. Rebuild and test
4. On next stable release, check if the issue is resolved and switch back to stable

### Switching Desktop Environments

1. Both KDE and GNOME are enabled simultaneously in the config
2. Log out from current session
3. At SDDM login screen, select the other session from the dropdown
4. Log back in
5. To disable a DE entirely: set its `.enable = false`, rebuild

### Switching the Default Session

```nix
# In configuration.nix:
services.displayManager.defaultSession = "plasma";  # or "gnome"
```
Rebuild and the next fresh login defaults to that session.

## Filesystem Recommendation

**Btrfs inside LUKS** with the following subvolume layout:

```
/dev/nvme0n1p1  -> /boot (EFI, FAT32, 512MB, unencrypted)
/dev/nvme0n1p2  -> LUKS2 container
  +-- btrfs filesystem
      |-- @        -> / (root)
      |-- @home    -> /home
      |-- @nix     -> /nix (largest, holds the Nix store)
      +-- @swap    -> /swap (swap file, optional)
```

**Why btrfs:**
- Snapshot support (rollback entire system state, not just NixOS generations)
- Transparent compression (zstd) saves SSD space, especially for `/nix/store`
- Subvolumes allow independent snapshot/backup of root vs home
- CoW semantics pair well with NixOS's immutable store

**Why separate @nix subvolume:**
- `/nix/store` is the largest directory on a NixOS system (20-50GB is normal)
- Compression is very effective on it (many similar files)
- Can exclude from home backups

## What Is and Is Not Declarative

### Fully Declarative (captured in config)
- All system packages and services
- Desktop environments (both KDE and GNOME) and display manager
- User packages (via Home Manager)
- Shell configuration (bashrc, aliases, PATH)
- Git configuration
- VS Code extensions and settings
- Flatpak remotes and installed apps (via nix-flatpak)
- Firewall rules
- Kernel parameters
- Boot loader configuration
- User accounts and groups
- Systemd services
- Garbage collection schedule and generation limits

### NOT Declarative (requires manual setup after install)
- Browser logins and bookmarks (unless using sync)
- Application credentials (Tailscale auth, Discord login, Google account, etc.)
- GNOME Online Accounts authentication (need to re-add Google/Microsoft accounts)
- SSH keys (copy from backup)
- GPG keys (copy from backup)
- Syncthing device pairings
- KDE Plasma customizations (wallpaper, panel layout, shortcuts) -- partially declarable via plasma-manager
- GNOME tweaks (extensions, appearance) -- partially declarable via dconf/Home Manager
- Flatpak app data and settings
- Docker images and volumes
- Ollama model downloads (need to re-pull)
- Project files in ~/projects (copy from backup)
- Browser extensions data (cookies, saved passwords)
- Email account configuration in Evolution/Betterbird/Kontact
